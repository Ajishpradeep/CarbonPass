### B2. Taiwan emission factors (put in `data/ef/`)
- **MOENV coefficient DB (`CFP_P_02` / data.gov.tw #28176).** Fields: `name`, `coe` (kgCO₂e), `unit`, `departmentname`, `announcementyear`.
  - **Do NOT test via the Swagger "Try it out"/Execute button** — its proxy 500s on these datasets. The "fire-key" field in Swagger is just where you paste your key (via **Authorize**); ignore it and call the URL directly.
  - **Canonical URL** (uppercase DataID, `api_key` as query param, **omit `language`**):
    `https://data.moenv.gov.tw/api/v2/CFP_P_02?limit=1000&sort=ImportDate%20desc&format=json&api_key=YOUR_KEY`
    - If 500: add `&offset=0`; then try `&language=zh`. Isolate key-vs-dataset with `aqx_p_432` (known-good) using the same key.
  - **Preferred for PoC (no key, reproducible):** download the static file from `https://data.moenv.gov.tw/dataset/detail/CFP_P_02` (資料預覽 → 下載篩選後資料 → JSON, or 歷史資料下載 → JSON/CSV), or the data.gov.tw mirror `https://data.gov.tw/dataset/28176`. Save as `data/ef/moenv_coefficients.json` and pin it (coefficients update irregularly). CSV mojibake → read as UTF-8.
  - Note: the live table may currently be thin/empty (platform changed its coefficient display logic 2026-02-10) — verify row count before relying on it; the static snapshot is the safe default.
- Grid EF note: **0.474 kgCO₂e/kWh (2024)** → `data/ef/grid_ef.json`.
- (Module 2) Taipower generation feeds #8931 (real-time) / #37331 (historical) — wire up a cached fetcher later; not needed for Sprint 0–3.

### B3. Nothing else is blocking
Real bills/invoices are **not** downloaded — they're generated synthetically (blueprint §9) until the pilot. e-GUI mock invoices use the open Turnkey lib: `https://github.com/phidiassj/TaiwanEInvoiceOpenAPI`.

## Part E — The Claude Code kickoff prompt (paste this)

> Open a new Claude Code session **inside the `carbonpass/` repo** (with `docs/` and `data/` prepared per Parts B–C), then paste:

```
You are the lead engineer building the CarbonPass PoC. Before writing any code, read these
docs in /docs in full and treat them as the source of truth:
  - /docs/10_poc_blueprint.md   (technical spec — architecture, stack, data sources, gaps)
  - /docs/09_master_proposal.md (product concept and rationale)
  - /docs/00_session_handoff.md (verified facts + a "kill-list" of claims that are WRONG — never reintroduce them)
  - /docs/11_setup_kickoff.md   (this setup, folder layout, and Definition of Done)

PROJECT IN ONE LINE: a local-first AI that turns a Taiwanese fastener SME's photographed
documents (Taipower bill, steel wire-rod e-invoices, production log) into a verifier-ready EU
CBAM "Communication Template for installations" (.xlsx) for CN 7318 goods, plus a default-vs-
actual buyer-cost screen. Module 2 (grid-aware scheduler) comes later; focus now is Module 1.

NON-NEGOTIABLE RULES (from the blueprint §2A — get these right):
  - CBAM is PER-PRODUCT-PER-YEAR, never per-shipment. Compute one SEE (tCO2e/tonne) per CN code
    per determination period (calendar year); the importer reuses it across shipments.
  - The output artifact is the producer→importer Communication Template, NOT the EU Registry
    declaration.
  - For fasteners (complex good), the dominant input is the PURCHASED STEEL precursor, then direct
    fuel, then production volumes, then electricity. Electricity is indirect and currently NOT in
    the CN 7318 certificate — don't overweight it.
  - Precursor SEE: use mill EPD if supplied, else the CBAM default value (data/cbam_official/
    default_values.xlsx) as conservative fallback, with the correct year mark-up (10/20/30%).
  - Every emitted figure must carry a source flag (actual/default) and a per-line uncertainty.
    The tool PREPARES verification; it never certifies.

STACK (already chosen — don't re-litigate): Python 3.11; Ollama serving qwen2.5vl:7b (vLLM +
guided_json for scale later); Docling for layout pre-pass; PaddleOCR/GOT-OCR2 as numeric backstop;
OR-Tools (MILP) + NumPy Monte-Carlo for allocation with uncertainty; openpyxl to fill the EU
template; FastAPI; LINE Messaging API for the front end. MiniCPM-V 2.6 is the only approved
alternative to benchmark against.

DO THIS SESSION, in order, committing after each step:
  1. Scaffold the repo exactly per /docs/11_setup_kickoff.md Part C (pyproject.toml, packages,
     .env.example, tests). Confirm deps install.
  2. Open data/cbam_official/template_examples/ (the "screws and nuts" workbook) and
     data/cbam_official/communication_template.xlsx. Transcribe the exact sheet list and the cells
     we must fill into schema/cbam_template_map.yaml. Capture the worked SEE numbers as the golden
     expected output in tests/golden/.
  3. Write scripts/make_mock_corpus.py to generate 3 firms: Taipower TOU bills (realistic layout),
     MIG 4.0 e-invoices via the open Turnkey format for steel/gas/plating, and per-CN-code
     production logs. Emit data/mock_corpus/{firm_a,firm_b,firm_c}/ + ground_truth.json (hand-
     computed SEE per firm).
  4. Build src/carbonpass/ingestion: Docling pre-pass → qwen2.5vl:7b (via Ollama) → activity_data
     JSON validated against schema/activity_data.schema.json. Run Firm A end-to-end.
  5. Build src/carbonpass/allocation (OR-Tools MILP + Monte-Carlo) and src/carbonpass/rules (SEE
     per IR 2025/2547 + default fallback + mark-up). Make tests/golden reproduce the screws & nuts
     numbers within tolerance.
  6. Build src/carbonpass/writer (openpyxl) to output a filled communication_template.xlsx for
     Firm A with actual/default/needs-attention flags, and src/carbonpass/costdelta for the buyer
     screen.
  7. Update README with exact run commands. Report status against the Definition of Done
     (/docs/11 Part D) and list any blockers.

Ask me before: changing the model/stack choices, adding a paid dependency, or anything that
implies per-shipment carbon logic. Otherwise proceed autonomously and keep commits small.
```
